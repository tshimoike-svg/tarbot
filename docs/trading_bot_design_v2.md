## 0. このボットの勝ち筋（最初に読むこと）

**前提：個人投資家が、GUIアプリ（kabuステーション）を経由するAPIで、ミリ秒の板の歪みを取りにいっても機関・HFTには勝てない。** だからこのボットは速度では戦わない。

代わりに、以下の3つが同時に成立する領域だけを狙う。

1. **容量制約ニッチ**：機関が「資金を投下しても旨味がない」ほど小さい歪み。彼らが無視する中小型株・特定時間帯。
2. **数秒〜数分スケールの統計的エッジ**：平均回帰（行き過ぎの反転）やイベント後の反応の偏り。速度ではなく「偏りの有無」で勝負する。
3. **コストを払わない執行**：手数料が無料になった今、頻回トレードで効くコストは**ビッド・アスク・スプレッド**。これを「払う」のではなく指値で「捕る」側に回る。

> **頻回トレードの本質的なトレードオフ（必読）**
> 1回あたりのエッジは極めて小さい。だから大数の法則で積み上げるために回数を増やす。
> しかし回数を増やすほどコスト（スプレッド・滑り・金利）が線形に膨らむ。
> **したがって「1トレードの期待値がコスト控除後でプラス」でない限り、回数を増やすほど負ける。** この一点が全ての設計判断の基準になる。

期待値の式（全戦略がこれを満たさなければ実装・稼働しない）：

```
E[1トレード] = 勝率 × 平均利益 − 敗率 × 平均損失 − 往復コスト
往復コスト ≒ スプレッド（実効） + 滑り + 信用コスト（持ち越し時）

稼働条件：E[1トレード] > 0  かつ  トレード数が統計的に十分（後述）
```

**正直な期待値の置き方**：個人のアルゴ・デイトレードの大半はコストに負けて退場する。この設計は「もしエッジが存在するならそれを規律よく抽出し、存在しないなら早く・安く見切る」ためのもので、利益を保証するものではない。資金は全損しても生活に影響しない範囲に限定すること。（本書は投資助言ではない。リスク管理の数値は必ず自分で検証・調整すること。）

## 1. プロジェクト概要

### 目的
日本株市場において、速度ではなく統計的偏りを源泉とするアルゴリズムで、感情を排除した規律ある頻回トレードを行い、コスト控除後でプラスの期待値を積み上げる。

### ねらい
- 機関・HFTが無視する容量制約ニッチに限定して戦う（正面衝突を避ける）
- 「バックテストで検証できる戦略」だけを実弾投入の前段に置き、検証不能な戦略はドライランでのみ運用する
- バックテスト → ドライラン → 最小ロット実取引 → 段階増資 という、各段で“見切り基準”を明確化した防御的アプローチでリスクを管理する

### 前提・制約
| 項目 | 内容 |
|------|------|
| 運用主体 | 個人投資家 |
| 証券会社 | 三菱UFJ eスマート証券（旧auカブコム証券） |
| 取引API | kabuステーションAPI（REST + WebSocket） |
| 取引種別 | 信用取引メイン（デイトレード／原則当日クローズ） |
| 対象市場 | 国内株式（流動性の確保された中小型株中心） |
| 開発言語 | Python 3.11以上 |
| 開発環境 | 自宅PC（Windows） |
| 本番環境 | さくらVPS（Windows必須に注意：後述） |
| 口座状況 | 開設手続き済み・審査中（信用取引口座も同時申請済み） |

## 4. kabuステーションAPI 仕様概要（事実修正済み）

- **動作方式**：kabuステーション（Windowsアプリ）が起動・ログイン状態でのみ使用可能
- **接続形式**：ローカルホスト経由（`http://localhost:18080`）
- **認証**：APIパスワード設定後にトークン取得
- **利用条件（修正）**：APIにはProfessionalプラン以上が必要。**初回ログイン時はProfessionalプランが翌々月第1営業日まで自動適用**され、以降は**適用開始日から翌月第1営業日まで**。継続にはほぼ毎月、適用条件を満たす取引（信用口座での取引等）が必要。**条件達成は容易だが“切れるとAPIが止まる”ため運用上のクリティカルパスとして月次監視する。**
- **手数料（確認済み）**：2026年5月18日以降、**SOR注文を選択した場合の**国内現物・信用の取引手数料が無料。
- **SOR強制の注意（新規）**：手数料無料はSOR注文選択が条件。SOR注文はPTS含む複数市場から有利な価格を探す仕組みで、執行先・タイミングが自分の指値意図と一致しない場面がある。**「APIからSORが指定可能か／SORを選ばないと手数料が発生するか／デイトレ戦略でSORが約定品質にどう影響するか」を口座開通前にサポートと公式ドキュメントで確定すること（→ §13で🔴タスク化）。**
- **信用コスト**：手数料無料化後も金利・貸株料は残る。デイトレ（当日返済）なら影響は小さいが、持ち越し・約定タイミング次第で効く。一般信用デイトレ系商品の金利体系も確認すること。

### 主要エンドポイント（予定）
```python
BASE_URL = "http://localhost:18080/kabusapi"

POST   /token                    # アクセストークン取得
GET    /board/{symbol}           # リアルタイム板情報（L1中心に使用）
GET    /symbol/{symbol}          # 銘柄情報
GET    /orders                   # 注文一覧
GET    /positions                # 保有建玉一覧
GET    /wallet/margin            # 信用余力
POST   /sendorder                # 株式注文（指値中心）
PUT    /cancelorder              # 注文取消（未約定指値の引き上げ/撤回に多用）
WS     /websocket                # リアルタイムPush配信
```

## 5. 差別化戦略（速度で戦わない版）

### 5-1. エッジの源泉
| 源泉 | 内容 | バックテスト可否 | 優先度 |
|------|------|----------------|--------|
| 日中平均回帰 | VWAP/移動平均からの短期的な行き過ぎの反転 | ◯（分足OHLCVで可能） | **最高** |
| 寄り後パターン | 寄り後30〜60分の統計的な反転/継続の偏り | ◯ | 高 |
| 引け・後場寄りの流動性イベント | 予測可能な時間帯の需給歪み | △（部分的に可能） | 中 |
| イベント反応（TDnet開示） | 開示直後の“初動の行き過ぎ”を分類して反応（速さではなく反応の偏りで勝つ） | △（開示タイムスタンプ＋分足が要る） | 中 |
| 相対強弱／簡易ペア | 同セクター内の相対的な歪み（マーケットニュートラル寄り） | ◯ | 中 |

> **板情報（オーダーフロー）・歩み値をシグナルの主軸にしない。** これらは(a)高品質な履歴データの入手が困難・高価、(b)自注文のマーケットインパクトを過去データで再現できない、ためバックテスト不能。使うなら「L1（最良気配の厚みの偏り）を数秒スケールの補助フィルタとして」に留め、検証はドライランのみとする。

### 5-2. なぜ機関に勝てる可能性があるのか（容量制約）
- 機関は1銘柄に億単位を入れるため、出来高の小さい中小型株の数bpの歪みは「拾っても運用規模に効かない」ので無視する。
- 個人は最小ロットで参加できるため、その小さな歪みを実質的に独占的に拾える。
- ただし**容量が小さい＝自分が利益を出せる資金量にも上限がある**。これは「規模拡大できない」という制約であって、最初は問題にならない。

### 5-3. 執行の差別化（コストを払わない）
- **指値中心**：スプレッドを払う成行ではなく、指値で約定を待ち、可能な限りスプレッドを捕る側に回る。
- **約定率と滑りを必ず計測**：指値は約定しないリスクがある。約定率・滑りをログ化し、戦略の期待値に織り込む（バックテストの楽観バイアス対策の核心）。
- **板が薄い時間帯・極端に流動性の低い銘柄を回避**：スプレッドが広い銘柄は手数料無料でもスプレッドで負ける。
- **過剰トレードの抑制**：1日の最大トレード回数に上限を設け、ノイズへの連続反応とバグ暴走の両方を防ぐ。

## 6. 取引戦略の方針

### 対象銘柄の選定基準
- 東証上場（プライム／スタンダードの中小型を中心に検討）
- **スプレッドが期待値に対して十分小さい**水準の流動性（最重要フィルタ）
- 出来高・売買代金が一定以上（指値が現実的に約定する流動性）
- 過度に低位・ボラ過大・仕手化銘柄は除外

### リスク管理ルール（再設計）
| ルール | 設定値（初期案） | 設計意図 |
|--------|---------------|----------|
| 1トレードの最大リスク | 資金の **0.5%** | 連敗してもドローダウンを生存可能域に抑える（固定%損切りより「1トレードで失う資金額」を固定） |
| 損切り | **ATRベースの相対値**（例：1.5×ATR） | 固定-2%はノイズで頻繁に刈られる。銘柄ボラに適応させる |
| 1銘柄あたり最大投資比率 | 資金の20%以内 | 集中リスク制限 |
| 1日の最大損失上限（サーキットブレーカー） | 資金の **3%**（v1の5%から保守化） | 到達したら当日全発注停止＋アラート |
| 1日の最大トレード回数 | 上限を設定（例：20〜40） | 過剰トレード・バグ暴走の防止 |
| 同時保有最大銘柄数 | 3〜5銘柄 | 監視可能性とリスク分散 |
| 期待値ゲート | 直近の実効期待値がコスト割れしたら戦略を自動停止 | エッジ消失の早期検知 |
| ポジション保有時間 | 原則当日中にクローズ | 持ち越し金利・オーバーナイトリスク回避 |

> すべての発注は例外なく `risk_manager.py` を通す。サーキットブレーカー・回数上限・期待値ゲートは発注前チェックとして強制する。

## 7. 検証フレームワーク（バックテスト可能/不能を分離）

### 7-1. 2つのトラックに分ける
**トラックA（バックテスト可能）**：日中平均回帰・寄り後パターン・相対強弱。
→ J-QuantsのOHLCVで過去検証 → ウォークフォワード → コスト控除後の期待値で合否判定。

**トラックB（バックテスト不能・ドライラン専用）**：イベント反応（TDnet）・L1補助フィルタ。
→ 過去再現できないため、リアルタイム・発注なしのドライランでシグナルの妥当性のみ評価。「シャープ>1.0」のような数値ゲートは適用せず、シグナルの方向性・頻度・コスト整合で判断する。

### 7-2. バックテストの落とし穴チェックリスト（evaluator.pyの必須要件）
- [ ] **ルックアヘッド・バイアス**：シグナル時点で未来の値を参照していないか
- [ ] **サバイバーシップ・バイアス**：上場廃止・併合・分割銘柄を含めて検証しているか（J-Quantsの廃止銘柄情報を使う）
- [ ] **取引コストの織り込み**：スプレッド・滑り・信用コストを控除した後の損益で評価しているか
- [ ] **滑り・約定率モデル**：指値の不約定、成行の滑りを保守的にモデル化しているか
- [ ] **マーケットインパクト**：自注文が薄い板を動かす前提を入れているか（中小型株では特に）
- [ ] **イン/アウトオブサンプル分離**：パラメータ最適化したデータと評価データを分けているか
- [ ] **統計的十分性**：トレード数が少なすぎないか（偶然の良成績を排除できる回数か）

### 7-3. 合否基準（再設計）
```
トラックA（バックテスト）：
  - コスト控除後の期待値 E[1トレード] > 0
  - ウォークフォワードの各区間で安定（特定期間依存でない）
  - 統計的に十分なトレード数（目安：数百回以上）
  - 最大ドローダウン < 15%
  ※「シャープ>1.0」はコスト控除後・アウトオブサンプルでのみ意味を持つ

トラックB（ドライラン）：
  - 2週間以上の安定稼働
  - シグナルの方向性・頻度・コスト整合が期待どおり
```

## 8. ダッシュボード仕様

- リアルタイムローソク足チャート（複数銘柄）＋VWAP・ATR・BBオーバーレイ
- 現在のポジション・含み損益

- 注文履歴・約定ログ・シグナル発生ログ
- 信用余力・Professionalプラン適用状況（残り日数の警告）

- 自動リフレッシュ：3秒間隔
- **ドライラン⇄本番の切替スイッチ**（既定はドライラン。本番切替時は確認ダイアログ＋ログ記録）

## 9. ML・LLM・強化学習の扱い（凍結と解凍条件）

v1ではPhase 2/3で機械学習・LLM感情分析・強化学習を積む計画だったが、**エッジが1つも確認できていない段階でMLを足すのは過学習装置を作るのと同義**のため、当面凍結する。

```
解凍条件（これを満たすまで一切実装しない）：
  トラックAのルールベース戦略が、コスト控除後・アウトオブサンプルで
  明確な正の期待値を、統計的に十分なトレード数で示したとき。

解凍後の優先順位：
  1) LightGBM等で「既存ルールの特徴量を拡張」する（ゼロからのML戦略は作らない）
  2) LLMによる開示文分類は、トラックBのイベント戦略が有望なら検討
  3) 強化学習による執行最適化は当面スコープ外（個人の初期PJでは費用対効果が低く、完成しないリスクが高い）
```

## 14. 重要な前提・既知のリスク（正直な開示）

- **大半の個人アルゴ・デイトレードはコストに負ける。** この設計は「エッジがあれば抽出し、なければ安く早く見切る」ためのもので、利益を保証しない。
- 投入資金は**全損しても生活に影響しない範囲**に限定する。
- **頻回トレードの最大の敵はスプレッドと滑り**であり、手数料無料はそれを解決しない。期待値計算でここを甘く見積もると本番で必ず負ける。
- バックテストの良成績は楽観バイアスの塊になりやすい（§7のチェックリストを必ず通す）。
- インフラ依存（kabuステーション常時起動・Professionalプラン維持・Windows必須）が単一障害点。監視と通知を設計に組み込む。
- 本書は技術設計のレビューであり、投資助言ではない。リスク管理の数値・銘柄選定は必ず自分で検証・判断すること。
